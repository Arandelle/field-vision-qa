# Image Q&A — WizyVision Technical Exercise

A stateless web app that lets a user upload an image, ask a natural-language question about it, and receive an answer generated by the Gemini API with full agentic reasoning, including intermediate Python code, execution results, and annotated images.

**Live URL:** `https://field-vision-qa.vercel.app/`

---

## How to Run Locally

**Prerequisites**
- Node.js 18+
- A [Gemini API key](https://aistudio.google.com/)

```bash
git clone https://github.com/Arandelle/field-vision-qa
cd field-vision-qa
npm install
```

Create a `.env` file in the project root:

```
GEMINI_API_KEY=your_key_here
```

Then start the dev server:

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000) in your browser.

The `.env` file is listed in `.gitignore` and is never committed. The API key is only read server-side inside the Next.js API routes — it is never exposed to the frontend bundle.

## Deployment

**Platform:** Vercel

**Why:** Zero-config Next.js deployment, no credit card required for the free tier, and environment variables are set through the dashboard, keeping the API key out of the repo entirely. The app is stateless with no database, so a serverless platform with a free tier is a natural fit.

## Architecture

```
Browser
  │
  ├── 1. User selects image
  │         │
  │   POST /api/upload (multipart/form-data)
  │         │
  │   Next.js Route (/app/api/upload/route.ts)
  │     - Forwards image bytes to Gemini Files API
  │     - Returns { fileUri, mimeType }
  │         │
  │   Gemini Files API
  │     - Stores file for 48 hours
  │     - Returns a stable file URI
  │
  ├── 2. User submits question (first turn or follow-up)
  │         │
  │   POST /api/ask (multipart/form-data)
  │     - fileUri + fileMime (not the image bytes)
  │     - question
  │     - history (full conversation turns, client-side)
  │     - x-request-id header (for end-to-end tracing)
  │         │
  │   Next.js Route (/app/api/ask/route.ts)
  │     - Reads GEMINI_API_KEY from process.env
  │     - Builds conversation contents with file_data reference
  │     - Re-attaches fileUri to first history turn on follow-ups
  │     - Calls streamGenerateContent?alt=sse
  │     - Forwards SSE chunks to browser as they arrive
  │         │
  │   Gemini gemini-3-flash-preview (Code Execution enabled)
  │     - Receives file URI + conversation history + question
  │     - Thinks, writes Python, executes it, annotates images
  │     - Streams chunks: thought → code → result → image → answer
  │         │
  │   Browser receives SSE stream
  │     - Each chunk parsed and rendered as a timeline step
  │     - Thought, code, result, annotated image, answer appear
  │       progressively as Gemini produces them
  │     - Completed turn appended to client-side history
  │       (including thought signatures for reasoning continuity)
```

## Key Decisions

**Next.js as a unified stack.** Rather than separating a backend (Express/FastAPI) from a frontend, Next.js API routes keep everything in one repo and one deployment unit. For a single-purpose stateless app this is simpler to reason about and deploy.

**Files API over inline base64.** The image is uploaded once when the user selects it, and only a URI string is sent on every subsequent request. This keeps the ask payload at ~0.6KB regardless of image size, compared to ~300KB+ with inline base64. It also means the image is not re-encoded or re-transmitted on follow-up turns — only the conversation history grows.

**Client-side conversation history.** Multi-turn context is stored in React state and sent with each request. The backend stays fully stateless — no session store, no database. The trade-off is that refreshing the page loses context, which is acceptable for this use case.

**Thought signatures preserved in history.** When storing the assistant's turn for multi-turn use, the full response parts including `thoughtSignature` fields are passed back to the client and re-sent on follow-up requests. This gives Gemini continuity in its reasoning state across turns rather than starting fresh each time.

**`streamGenerateContent` for true streaming.** The ask route calls `streamGenerateContent?alt=sse` and forwards Gemini's chunks directly to the browser as SSE events. Each agentic step — thought, code, execution result, annotated image, answer — appears in the UI as Gemini produces it, not after the full response completes.

**Client-side and server-side validation both.** File size is checked in the browser before the request is sent (fast feedback) and again on the server (security). The server is the authority; the client check is a convenience.

**Plain `fetch` to the Gemini REST API.** The REST endpoint is called directly rather than using the `@google/generative-ai` SDK to keep dependencies minimal and make the request/response contract explicit. Every field in the request body is intentional and visible.

**Structured logging with request IDs.** Each request generates a UUID via `crypto.randomUUID()` (no dependencies), passed as `x-request-id` header, logged at every meaningful step server-side as structured JSON, and displayed in the UI. This makes it straightforward to correlate a specific UI response with its backend log entries.

## What I Would Do With More Time

- **True token-level streaming:** The current implementation streams parsed parts as SSE events, but each part is still a complete chunk from Gemini. Token-level streaming would require handling partial text chunks and assembling them progressively in the UI.
- **Files API expiry handling:** Uploaded files expire after 48 hours. A production version would need to detect expired URIs and prompt the user to re-upload rather than failing silently.
- **Rate limiting:** Basic rate limiting on the API routes to prevent abuse and control cost.
- **Image preprocessing:** Optional resize before upload for very large images to reduce Gemini token usage and improve latency.
- **AI-generated prompt suggestions:** Use a lightweight Gemini call on image select to suggest relevant questions based on what is actually in the image, rather than static preset buttons.
