# Image Q&A — WizyVision Technical Exercise

A stateless web app that lets a user upload an image, ask a natural-language question about it, and receive an answer generated by the Gemini API.

**Live URL:** `https://field-vision-qa.vercel.app/`

---

## How to Run Locally

### Prerequisites

- Node.js 18+
- A [Gemini API key](https://aistudio.google.com)

### Steps

```bash
git clone https://github.com/Arandelle/field-vision-qa
cd field-vision-qa
npm install
```

Create a `.env.local` file in the project root:

```
GEMINI_API_KEY=your_key_here
```

Then start the dev server:

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000) in your browser.

> The `.env.local` file is listed in `.gitignore` and is never committed. The API key is only read server-side inside the Next.js API route — it is never exposed to the frontend bundle.

---

## Deployment

**Platform chosen:** `Vercel`

**Why:** `Vercel has zero-config Next.js deployment, no credit card required for the free tier, and environment variables are set through the dashboard — keeping the API key out of the repo entirely.` 

The app is stateless and has no database, so a serverless or container-based platform with a free tier is a natural fit. The Gemini API key is stored as an environment variable in the platform's dashboard, not in the repository.

## Architecture

```
Browser
  │
  ├── Uploads image (max 5 MB, validated client-side before submission)
  └── Sends POST /api/ask (multipart/form-data)
            │
    Next.js API Route (/app/api/ask/route.ts)
      - Re-validates file size server-side
      - Reads GEMINI_API_KEY from process.env (never touches the client)
      - Converts image to base64
      - Calls Gemini REST API with image + question
      - Returns { answer } or { error }
            │
    Gemini gemini-3-flash-preview
      - Receives inline image data + text prompt
      - Returns a text answer
```

### Key decisions

**Next.js as a unified stack.** Rather than separating a backend (Express/FastAPI) from a frontend, Next.js API routes let me keep everything in one repo and one deployment unit. For a single-purpose stateless app this is simpler to reason about and deploy.

**Client-side and server-side validation both.** File size is checked in the browser before the request is sent (fast feedback) and again on the server (security). The server is the authority; the client check is a convenience.

**Plain `fetch` to the Gemini REST API.** I called the REST endpoint directly rather than using the `@google/generative-ai` SDK to keep dependencies minimal and make the request/response contract explicit. Every field in the request body is intentional and visible.

## What I Would Do With More Time

- **Tier 2 — Agentic Vision:** Enable Code Execution on Gemini 3 Flash and surface the Think → Act → Observe steps, intermediate images, and executed Python code as a readable timeline in the UI.
- **Streaming:** Stream agentic steps as they arrive rather than waiting for the full response, which would significantly improve perceived latency for longer reasoning chains.
- **Multi-turn:** Keep the image in memory (or re-attach it) so the user can ask follow-up questions without re-uploading.
- **Preset prompt templates:** Buttons like "Read serial number" or "Count items" that fill in a tuned prompt, matching the field operations use case described in the brief.
- **AI-generated prompt suggestions:** Use Gemini to analyze the uploaded image and suggest relevant questions (e.g., "Count items", "Read serial number", "Check safety compliance"), helping users who may not know what to ask.
- **Structured logging with request IDs:** Attach a unique request ID to each request and log it across the API lifecycle to improve debugging and traceability.
- **Image preprocessing:** Resize or compress images before sending to Gemini to reduce latency and token usage.
- **Rate limiting:** Add basic rate limiting to prevent abuse of the API endpoint and control cost.
